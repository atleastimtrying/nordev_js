JS Structure

  • A series of simple refactors showing how I tidy up legacy code.
  • These are in no particular order
  • They are a personal approach, I'm always trying to improve these.

0: .js files

  • tidies up your html
  • no more digging through templates
  • gives you clear structure
  • use multiple files and join them together (grunt et al)
  • helps you think abour load order and dependencies

1: Namespacing

  • namespacing prevents libarary clashes
  • it's a simple step that can tidy your entire codebase
  • it makes your code distict from libraries
  • use closures to seperate your code.

2: Singletons

  • in js the singleton pattern can be a simple object literal
  • it helps re enforce your namespacing and group similar functions together
  • it can give you an idea as to the kinds of structures you want to expose to your user

3: pseudo class structures

  • you can emulate classes in js
  • use classes for objects with specific behavior
  • dealing with collections is neater with an array of instances

4: discrete functions

  • not all functions need to be public
  • only expose the functions needed outside of the context
  • using var inside a closure keeps functions discrete
  • using this. inside an instance exposes functions explicitly

5: good naming

  • all names should be clear
  • use js idioms or don't. You know your audience
  • careful using capitals at the start as they imply a class.
  • use a sensible consistent glossary
  • never have a function name describing two actions

6: loads of tiny functions

  • javascript gives us anonymous functions
  • use them at your peril
  • small named functions with one purpose are testable
  • they are also more legible
  • combine time sensitive functions into procedural groups
  • write more functions
